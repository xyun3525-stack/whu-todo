# Gamified Campus Todo V2 设计文档

- 日期：2026-04-24
- 状态：已完成设计确认，待进入实现计划
- 适用仓库：`D:\whu-todo\todo_gamification - 副本`

## 1. 背景

当前项目是一个基于 `tkinter` 的本地单机 Todo 应用，已经具备以下闭环：

- 添加任务
- 完成任务获得 XP
- 随机掉落建筑
- 在山体网格上摆放建筑
- 升级后扩地或开垦

现有实现已经形成了可玩的基础，但仍然更接近一个有趣的原型，而不是可长期使用的产品。主要问题如下：

- 任务系统较轻，适合演示，不足以支持日常长期使用
- 奖励系统以 XP 和随机掉落为主，成长目标不够稳定
- 山体页已有视觉特色，但经营深度不足
- `gui.py` 已承担过多界面与交互职责，后续扩展风险较高
- 存储仍是简单的 `data.json` 结构，缺少版本迁移与扩展边界

## 2. 产品目标

V2 的目标不是把项目改成通用的冷冰冰效率工具，而是将其升级为：

**一个以校园成长为主题、可长期使用的游戏化待办应用。**

产品需要同时满足两件事：

1. 让用户真的愿意把日常任务放进来管理
2. 让用户在完成任务时持续感受到校园建设与个人成长的推进感

## 3. 已确认方向

本轮设计已经确认以下决策：

- 产品方向选择 `通用 gamified todo` 的底层思路，但默认主题保留校园化
- V2 不取消“珞珈山/校园建设”这条叙事线
- 后续允许提供轻量自定义能力，例如修改山体名称、校园名称或主题文案
- 第一版不追求复杂的多主题系统，优先保留当前校园辨识度
- 产品重点是“实用性 + 成长反馈并重”，不是单纯追求抽卡感或装饰感

## 4. V2 产品定位

### 4.1 一句话定位

把真实完成的任务，转化为校园建设与成长进度的游戏化待办应用。

### 4.2 目标体验

用户打开应用后，应该能立即知道三件事：

- 我今天该做什么
- 我做完之后会得到什么
- 我的校园还差多少能进入下一阶段

### 4.3 设计原则

- 先让任务管理够用，再增加游戏层
- 奖励必须有确定收益，随机只负责惊喜，不负责核心价值
- 主题保留校园风格，但底层规则尽量通用
- 长期目标要稳定可见，避免纯随机成长
- 系统要支持渐进扩展，避免一次性大重构

## 5. 核心循环

V2 的核心循环定义为：

`规划今天 -> 完成任务 -> 获得成长收益 -> 推进校园建设 -> 解锁长期目标`

与当前版本相比，核心变化有三点：

- 从“完成任务后只看 XP/掉落”升级为“完成任务后同时推进多个进度条”
- 从“随机掉落驱动”升级为“固定成长 + 随机惊喜”
- 从“单个标签页体验”升级为“首页驱动的日常使用体验”

## 6. 信息架构

V2 推荐采用以下主要页面结构：

- `今日`
- `任务`
- `校园`
- `成长`
- `设置`

### 6.1 今日页

今日页是主入口，优先级最高，负责提供当日行动感。

展示内容：

- 今日待办数量
- 已完成数量
- 当前连续完成天数
- 今日目标进度
- 当前校园等级 / 建设进度
- 快速添加任务入口

### 6.2 任务页

任务页负责完整管理任务，强调可用性，不做重型项目管理。

支持能力：

- 新增任务
- 编辑任务
- 删除任务
- 标记完成
- 按状态筛选
- 按优先级/截止时间查看
- 今天 / 计划中 / 已完成三级视图

### 6.3 校园页

校园页保留山体建造特色，但从“摆放界面”升级为“经营结果视图”。

展示内容：

- 山体/校园主视图
- 区域解锁状态
- 地块使用情况
- 建筑摆放与建筑效果
- 校园繁荣度或建设度
- 山体名称编辑入口

### 6.4 成长页

成长页负责承载长期目标与成就感。

展示内容：

- 玩家等级与 XP 进度
- 连击天数
- 周目标进度
- 建筑图鉴
- 最近获得奖励
- 阶段性里程碑

## 7. 任务系统设计

V2 的任务系统采用“够用但不臃肿”的策略，重点服务日常使用。

### 7.1 Task 模型建议新增字段

- `tags`
- `estimated_minutes`
- `subtasks`
- `repeat_rule`
- `completed_at`
- `scheduled_for_today`

### 7.2 任务视图规则

- `今天`：今日计划内任务，强调执行
- `计划中`：未来要做但不一定今天完成
- `已完成`：历史记录，用于复盘与奖励回顾

### 7.3 任务设计边界

以下能力先不纳入 V2 首版范围：

- 团队协作
- 多人共享任务
- 复杂项目树
- 云同步

## 8. 奖励与成长系统设计

### 8.1 奖励原则

奖励系统采用：

**固定成长 + 轻随机惊喜 + 稳定长期目标**

这意味着每次完成任务都必须产生确定收益，而随机掉落只作为额外情绪价值。

### 8.2 结算流程

推荐的任务结算流程如下：

`任务完成 -> 计算基础成长值 -> 应用修正 -> 发放固定奖励 -> 触发随机奖励 -> 推进日/周目标`

### 8.3 固定奖励

固定奖励至少包含：

- XP
- 校园建设资源
- 周目标推进值

可选扩展奖励：

- 金币
- 图鉴碎片
- 连击奖励点数

### 8.4 随机奖励

随机奖励保留建筑掉落机制，但需要从“纯随机核心收益”改为“附加奖励”。

规则建议：

- 每次完成任务都可触发掉落判定
- 掉落结果可以是完整建筑或建筑碎片
- 连击、周目标和特定建筑可提高稀有掉落率
- 提供稀有保底，避免长期脸黑

### 8.5 连击系统

连击应当是激励，而不是惩罚源。

设计要求：

- 连续完成天数提供轻量增益
- 周维度优先于日维度
- 漏一天时不应完全清空全部进度感
- 可后续加入“缓冲天”或“连击保护”机制

### 8.6 反刷机制

为了防止通过拆分超小任务无限刷奖励，建议加入以下限制：

- 同类超短任务连续完成时奖励边际递减
- 重复任务收益低于真正推进型任务
- 每日高收益任务数量有上限，后续趋于平稳
- 任务规模过碎时不允许线性倍增收益

## 9. 校园经营系统设计

### 9.1 长期成长轴

校园成长至少包含以下三条长期进度：

- `校园等级`
- `区域建设度`
- `图鉴完成度`

它们分别负责：

- 校园等级：决定阶段与系统解锁
- 区域建设度：决定地图、地块和新区域开放
- 图鉴完成度：提供收藏与长期目标感

### 9.2 建筑分类

建筑建议划分为三类：

- `功能建筑`
- `地标建筑`
- `装饰建筑`

说明如下：

- 功能建筑：提供 XP、掉落率、周目标收益等加成
- 地标建筑：提供里程碑性质的稀有效果
- 装饰建筑：以美观、收藏和少量繁荣度为主

### 9.3 建筑效果首版范围

V2 首版建议只保留少量清晰效果：

- XP 加成
- 稀有掉落加成
- 周目标奖励加成
- 连击保护或连击收益加成

这样更利于平衡，也便于用户理解。

### 9.4 名称自定义

在不破坏校园主题的前提下，允许后续逐步开放轻量自定义：

- 山体名称可修改
- 校园名称可修改
- 个别区域名称可修改

V2 首版只要求支持“山体名称可编辑”。

## 10. 技术落地方案

### 10.1 现有结构保留

保留当前主结构：

- `main.py`
- `models.py`
- `game_logic.py`
- `storage.py`
- `gui.py`

但逐步拆清职责，而不是继续把所有能力堆到单文件里。

### 10.2 数据模型演进

推荐新增或拆分的核心模型：

- `Task`
- `Player`
- `Campus`
- `Collection`

说明如下：

- `Task`：任务属性与执行状态
- `Player`：等级、XP、连击、基础资源
- `Campus`：山体、区域、地块、命名、繁荣度
- `Collection`：建筑图鉴、碎片、保底计数

### 10.3 逻辑层拆分

`game_logic.py` 内部逐步演进为以下职责：

- `TaskService`
- `RewardEngine`
- `ProgressionService`
- `BuildingService`

即使首版仍放在同一个文件中，也应按职责分方法与区域组织，避免 `complete_task()` 持续膨胀。

### 10.4 存储层演进

继续使用 JSON 本地存储，不强制升级数据库。

但需要引入：

- 顶层 `version` 字段
- 更明确的顶层存储结构

推荐结构：

- `version`
- `player`
- `tasks`
- `campus`
- `collection`
- `settings`

### 10.5 界面层拆分

`gui.py` 需要在 V2 中逐步拆分为更小的模块。

推荐结构：

- `main_window.py`
- `tabs/today_tab.py`
- `tabs/tasks_tab.py`
- `tabs/campus_tab.py`
- `tabs/growth_tab.py`
- `dialogs/task_dialog.py`
- `dialogs/reward_dialog.py`

### 10.6 数据流约束

推荐统一为如下数据流：

`UI 操作 -> 逻辑服务 -> 更新模型 -> 持久化 -> 刷新视图`

不建议让不同标签页彼此直接修改对方状态，以免后续出现刷新和一致性问题。

## 11. 错误处理与兼容性

V2 必须提前考虑旧存档和异常数据。

最低要求：

- JSON 读取失败时回退默认数据
- 旧版本字段缺失时提供默认值
- 未知建筑 ID 跳过处理但不崩溃
- 奖励结算失败时保证任务状态与存档一致

## 12. 测试重点

优先补逻辑测试，而不是先做 GUI 自动化。

建议覆盖：

- 奖励计算
- 连击和保底逻辑
- 建筑效果叠加
- 存档迁移
- 旧数据兼容

验证重点是：

- 算得对
- 存得住
- 升级后不炸旧档

## 13. 分阶段实施建议

### Phase 1：打基础

- 扩展数据模型
- 重构存储结构
- 增加版本字段和迁移逻辑
- 重写奖励结算主流程

### Phase 2：补核心体验

- 新增今日页
- 增强任务页
- 增加成长页
- 让任务完成结果同步刷新多个页面

### Phase 3：深化校园经营

- 建筑分类与效果整理
- 区域建设度与长期目标接入
- 图鉴系统接入
- 山体名称可编辑

### Phase 4：打磨与扩展

- 提升反馈动画和结算体验
- 打磨统计展示
- 预留未来主题自定义能力

## 14. V2 首版必须交付

实现计划必须至少覆盖以下可见成果：

- `今日` 页面可用，支持查看今日任务与当日进度
- 任务支持新增、编辑、完成、删除与基础筛选
- 任务完成后同时结算固定奖励和随机奖励
- 校园页可查看区域/地块/建筑状态
- 成长页可查看等级、连击、周目标和图鉴基础信息
- 存档结构具备版本字段和迁移能力
- 山体名称支持编辑

如果某项设计无法进入首版，应优先砍扩展项，而不是砍掉上述主链路。

## 15. 非目标

以下内容明确不属于 V2 首版范围：

- 多端实时同步
- 账号系统
- 多人协作
- 在线排行榜
- 复杂经济系统
- 大量数值职业分支

## 16. 成功标准

如果 V2 达到以下状态，可视为方向成功：

- 用户能把它当成日常待办工具连续使用
- 完成任务时能持续感受到校园成长推进
- 奖励系统不会被简单拆任务轻易刷爆
- 校园页不只是展示，而是有长期经营目标
- 当前代码结构能继续扩展，而不会被单文件拖垮

## 17. 结论

V2 的本质升级不是“多加几个功能”，而是把当前项目从一个有趣的校园化 Todo 原型，升级成一个真正可持续使用的校园成长型产品。

本设计选择的路线是：

- 保留校园主题
- 强化任务实用性
- 用稳定成长目标替代纯随机驱动
- 采用渐进式技术演进，避免推倒重写

在此基础上，后续实现计划应优先围绕 `数据模型 -> 奖励主流程 -> 页面结构 -> 校园经营深化` 展开。
